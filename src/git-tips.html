Git Tips

Создает новый репозиторий Git и указывает на использование ветки с названием "main" в качестве основной ветки (вместо стандартной "master"). Это часто используется в контексте смены стандартного названия основной ветки с "master" на "main" в соответствии с современными стандартами и рекомендациями.

git init -b main

Добавить все измененные и новые файлы в рабочем каталоге в индекс для последующего коммита.

git add .

Команда git commit в Git используется для фиксации (зафиксирования) изменений, внесенных в файлы вашего проекта. Коммит создает новую "точку сохранения" в истории вашего проекта, которая позволяет вам вернуться к состоянию проекта на определенном этапе разработки или сравнить изменения между коммитами.

git commit -m "Initial commit"

Список опций:

  • -a или --all: указывает Git'у автоматически включить все измененные и удаленные файлы в вашем рабочем каталоге, которые уже отслеживаются Git'ом, в коммит. Это позволяет вам пропустить этап добавления файлов git add перед фиксацией, если вы уверены, что ваши изменения должны быть включены в коммит
  • -m или --message: Используется для добавления сообщения о коммите (commit message) к вашему коммиту. Сообщение коммита представляет собой краткое описание изменений, внесенных в рамках этого коммита. Обычно это должно быть описание того, что было изменено или исправлено. Если вы опустите этот флаг, Git откроет редактор (обычно Vim или Nano), чтобы вы могли ввести сообщение о коммите в отдельном окне.

Добавляет удаленный репозиторий с именем "origin" и связывает его с указанным URL-адресом удаленного репозитория.

git remote add origin <URL_удаленного_репозитория>

Изменяет URL-адрес удаленного репозитория "origin" на указанный URL, включая ваш токен доступа, имя пользователя и имя репозитория.

git remote set-url origin https://access_token@github.com/username/repository.git

Извлекает изменения из удаленной ветки "main" в ваш локальный репозиторий и объединяет их с текущей локальной веткой.

git pull origin main

Используется для отправки (push) изменений из локальной ветки main в удаленный репозиторий с именем origin.
Опция -u или --set-upstream устанавливает текущую локальную ветку для отслеживания (tracking) ветки main на удаленном репозитории origin. Это означает, что в дальнейшем при использовании команды git pull и git push, Git будет знать, с какой веткой на удаленном репозитории связывать текущую локальную ветку:

git push -u origin main

Используется для отправки (загрузки) локальных изменений в удаленный репозиторий. При использовании этой команды Git отправляет все изменения, которые были закоммичены в вашей текущей локальной ветке, на удаленный сервер.

git push

Выводит список всех настроек Git в вашем локальном репозитории. Эти настройки могут включать данные пользователя, настройки форматирования, конфигурацию удаленных репозиториев и многое другое.

git config --list

Выводит историю коммитов в формате, где каждый коммит представлен сокращенным идентификатором, именем автора, и временем с момента его создания, а также отображает графическое представление ветвлений и объединений коммитов.

git log --pretty=format:"%h - %an, %ar" --graph
git log --pretty=oneline

Запускает интерактивное перебазирование для последних двух коммитов на вашей текущей ветке. Эта команда позволяет переписывать историю коммитов, объединяя, изменяя порядок, сливая или редактируя коммиты перед их применением к ветке.

git rebase -i HEAD~2

Например, если вы хотите объединить второй коммит с первым, измените pick на squash для второго коммита. Коммиты указаны в порядке возрастания времени создания (самый нижний -- самый свежий).

pick bcdca61    # First commit
squash e0ca8b9  # The last commit

После объединения коммитов, если вы хотите отправить изменения в удаленный репозиторий, вам придется использовать команду:

git push --force

Создает новую ветку с указанным именем и переключает текущую рабочую область на эту ветку. Это удобно, когда ты хочешь создать новую ветку и сразу начать работу в ней.

git checkout -b <название_новой_ветки>

Для вывода списка веток, которые существуют в вашем репозитории, но не были слиты (не объединены) с текущей веткой. Это полезно для определения веток, в которых содержится непринятый код или незавершенная работа.

git branch --no-merged

Пример более подробного вывода. Краткий хэш, имя и дата автора:

git for-each-ref --format='%(objectname:short) %(refname:short) %(authordate:relative)' refs/heads --no-merged

Если каталог не игнорируется, возможно, он уже был добавлен в репозиторий до добавления в .gitignore. В таком случае Git будет следить за изменениями в этом каталоге, игнорируя только изменения в .gitignore после того, как они были применены.

Чтобы исправить это, вам нужно удалить каталог из репозитория и зафиксировать это изменение. Для этого выполните следующие команды в терминале (поменяйте "directory_name" на имя вашего каталога):

git rm -r --cached directory_name
git commit -m "Remove directory_name from the repository"
git push origin master (или имя вашей ветки)

После этого Git перестанет отслеживать изменения в этом каталоге.

Добавить незначительные правки в последний коммит и, при этом, избежать засорения истории сообщениями вида «Ой, забыл добавить файл» или «Исправление грамматической ошибки».

Пример:

git commit -m 'Initial commit'  # Сделали commit
git add forgotten_file  # Захотели добавить файл

После манипуляций:

git commit --amend

Очень важно понимать, что когда вы вносите правки в последний коммит, вы не столько исправляете его, сколько заменяете новым, который полностью его перезаписывает. В результате всё выглядит так, будто первоначальный коммит никогда не существовал, а так же он больше не появится в истории вашего репозитория.

Для исключения файла из индекса:

git reset HEAD <file_name>

git restore --staged <file_name>

Чтобы отменить изменения в файле и вернуть его к состоянию, которое было в последнем коммите

git checkout -- <file_name>

git restore <file_name>

Важно понимать, что все локальные изменения в файле пропадут — Git просто заменит его версией из последнего коммита. Ни в коем случае не используйте эту команду, если вы не уверены, что изменения в файле вам не нужны.

Удаление ветки в Git. Обратите внимание, что удаление ветки является необратимым действием, поэтому убедитесь, что вы действительно хотите удалить ветку, прежде чем выполнить эту команду.

git branch -d feature

Список опций:

  • -d: если ветка, которую вы пытаетесь удалить, не была полностью слита с текущей веткой, Git выдаст предупреждение и не удалит ветку.
  • -D: если вы хотите удалить ветку, игнорируя эти предупреждения, используйте эту опцию.

Создание аннотированного тега в Git.

git tag -a v1.4 -m "my version 1.4"

Список опций:

  • -a: указывает, что тэг аннотированный.
  • -m: задаёт сообщение, которое будет храниться вместе с тегом. Если не указать сообщение, то Git запустит редактор, чтобы вы смогли его ввести.

Создание легковесного тега в Git.

git tag v1.4-lw

Удаление тега в Git.

git tag -d v1.4-lw

Удаление тега из внешнего репозитория.

git push  :refs/tags/<tagname>

git push origin --delete <tagname>

Добавить тег к существующиму камиту.

Пример:

git tag -a v1.2 9fceb02

Переход на тег.

Пример:

git checkout v2.0.0

Обмен тегами.

По умолчанию, команда git push не отправляет теги на удалённые сервера. После создания теги нужно отправлять явно на удалённый сервер.

git push origin v1.5

Для всех тэгов:

git push origin --tags

Перебазирование текущей ветки.

В результате выполнения этих команд ваша ветка main будет содержать изменения, внесённые веткой double_feature, и ветка double_feature будет удалена. 

  • git rebase main

    Эта команда выполняет перебазирование текущей ветки на ветку main. В результате история коммитов вашей текущей ветки будет переписана поверх истории коммитов из ветки main. 

  • git checkout main

    Эта команда переключает вас на ветку main. Теперь ваш рабочий каталог содержит содержимое последнего коммита в ветке main. 

  • git merge double_feature

    Здесь вы сливаете ветку main в текущую ветку main. Эта команда объединяет изменения из ветки double_feature с изменениями в вашей ветке main. 

  • git branch -d double_feature

    Эта команда удаляет ветку double_feature. После успешного слияния её изменений в ветку main она становится ненужной и может быть безопасно удалена.
